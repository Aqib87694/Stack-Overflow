SECTION 1 — HOW TO RUN THE PROJECT
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Step 1 → Open Command Prompt or PowerShell
Step 2 → Navigate into the project folder:
         cd path\to\so_dashboard

Step 3 → Install all libraries:
         python -m pip install -r requirements.txt

Step 4 → Launch the dashboard:
         python -m streamlit run app.py

Step 5 → A browser tab opens automatically at:
         http://localhost:8501

NOTE: First load takes ~10 seconds because Pandas reads 89,184 rows.
      After that, all filter interactions are instant (Streamlit cache).

SECTION 2 — PYTHON COMMANDS EXPLAINED (filters.py)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Every command in filters.py and why it was used:

──────────────────────────────────────────────────────────────────────────────
COMMAND: pd.read_csv(filepath, low_memory=False)
WHY: Loads the 152MB CSV file into a Pandas DataFrame.
     low_memory=False is required because some columns have mixed data types
     (numbers stored as strings) — without this flag, Pandas guesses types
     in chunks and gives DtypeWarning errors.

COMMAND: df.drop(columns=[...])
WHY: Removes columns that are irrelevant to our analysis targets
     (salary, experience, AI adoption, language, demographics).
     Dropping unnecessary columns reduces RAM usage from ~1.2GB to ~400MB
     and speeds up every subsequent operation.

COMMAND: pd.to_numeric(df["YearsCode"], errors="coerce")
WHY: The YearsCode column contains text like "More than 50 years" alongside
     numbers. errors="coerce" turns unreadable values into NaN (missing)
     instead of crashing. This is essential for arithmetic operations later.

COMMAND: df.loc[df["ConvertedCompYearly"] > 600_000, "ConvertedCompYearly"] = np.nan
WHY: Removes extreme salary outliers (some respondents entered $70 million+).
     These are data entry errors. Keeping them would pull averages and charts
     wildly off-scale, hiding meaningful patterns in the $0–$300K range.

COMMAND: df["Employment"].str.split(";").str[0].fillna("Unknown")
WHY: Employment is a multi-select column where respondents could choose
     multiple values separated by semicolons, e.g.
     "Employed, full-time;Independent contractor...".
     .str.split(";")  → splits the string into a list
     .str[0]          → takes only the PRIMARY employment type
     .fillna("Unknown")→ replaces missing values with "Unknown"

COMMAND: ed_map = {...}  then df["EdLevel"].map(ed_map)
WHY: Education level strings in the raw data are very long, e.g.
     "Bachelor's degree (B.A., B.S., B.Eng., etc.)".
     .map() replaces them with short labels like "Bachelor's" so they
     fit neatly on chart axes without overlapping.

COMMAND: def salary_band(s): ... then df["ConvertedCompYearly"].apply(salary_band)
WHY: Creates a categorical salary band column ($30K–$60K, $60K–$100K, etc.)
     from the continuous salary number.
     .apply() runs our custom function row-by-row.
     This allows grouping and colour-coding by salary tier in charts.

COMMAND: df["ExpBucket"] = df["YearsCodePro"].apply(exp_bucket)
WHY: Groups continuous years-of-experience into readable buckets
     (0–1 yrs, 2–4 yrs, 5–9 yrs, etc.) for charts that need categories.

COMMAND: df.reset_index(drop=True)
WHY: After filtering rows, index numbers have gaps (e.g. 0, 5, 12...).
     reset_index creates a clean 0, 1, 2... index.
     drop=True means the old index is discarded, not added as a column.

──────────────────────────────────────────────────────────────────────────────
apply_filters() — why each filter works this way:

COMMAND: f[f["Age"].isin(selected_age)]
WHY: Boolean indexing — keeps only rows where Age is in the selected list.
     .isin() is more efficient than multiple == conditions chained with |.

COMMAND: f["ConvertedCompYearly"].isna() | ((f["ConvertedCompYearly"] >= lo) & ...)
WHY: For the salary range slider, we keep rows where salary is:
     (a) missing/NaN — so respondents without salary data still appear
         in non-salary charts like language and AI adoption, OR
     (b) within the selected range.
     Without the .isna() check, the salary slider would hide 46% of
     respondents from ALL charts just because they didn't report salary.

COMMAND: f["Country"].str.lower().str.contains(term, na=False)
WHY: Free-text search. .str.lower() makes search case-insensitive.
     .str.contains(term) finds partial matches (so "ind" matches "India").
     na=False means NaN countries don't raise errors.

──────────────────────────────────────────────────────────────────────────────
compute_kpis() — the 10 summary numbers:

COMMAND: df["LanguageHaveWorkedWith"].str.split(";").explode().value_counts().index[0]
WHY: Top language calculation.
     .str.split(";") → turns "JavaScript;Python;SQL" into a list
     .explode()       → one row per language (expands the lists)
     .value_counts()  → counts how often each language appears
     .index[0]        → picks the most frequent one

SECTION 3 — PYTHON COMMANDS EXPLAINED (charts.py)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

──────────────────────────────────────────────────────────────────────────────
COMMAND: matplotlib.use("Agg")
WHY: Forces Matplotlib to use a non-interactive backend (Agg = Anti-Grain
     Geometry renderer). Without this, Matplotlib tries to open GUI windows
     on every render, which crashes inside Streamlit's server environment.

COMMAND: fig.patch.set_facecolor(BG) then ax.set_facecolor(SURF)
WHY: fig.patch is the OUTER canvas (the entire figure area).
     ax is the INNER plotting area.
     Both need dark colours separately — setting one does not affect the other.

COMMAND: ax.set_axisbelow(True)
WHY: Puts grid lines BEHIND the data (bars, points, lines).
     Without this, grid lines draw on top of bars, making charts messy.

COMMAND: ax.yaxis.set_major_formatter(mticker.FuncFormatter(_fmt_k))
WHY: Replaces raw numbers like "100000" with readable "$100K" labels on axes.
     FuncFormatter takes a custom function and applies it to every tick label.

COMMAND: ax.fill_between(x, pcts, alpha=0.12, color=NEON)
WHY: Draws a shaded area under the line chart.
     alpha=0.12 makes it very transparent so it looks like a subtle glow,
     not a solid block — adds depth without hiding the line.

COMMAND: df.sample(min(4000, len(df)), random_state=42)
WHY: The scatter plot with 89K points would be a black blob and
     take 10+ seconds to render. We randomly sample 4000 points —
     enough to show the full pattern clearly.
     random_state=42 makes sampling reproducible (same points every time).

COMMAND: np.polyfit(x, y, 1) then np.poly1d(z)(xs)
WHY: Fits a 1st-degree polynomial (straight line) through the scatter data.
     np.polyfit returns [slope, intercept].
     np.poly1d creates a callable function from those coefficients.
     Plotting this shows the overall salary trend with experience.

COMMAND: sns.heatmap(..., mask=np.triu(..., k=1))
WHY: mask=np.triu hides the upper triangle of the correlation matrix.
     The matrix is symmetric (correlation of A vs B = B vs A), so showing
     both triangles is redundant. Showing only the lower triangle
     makes the chart cleaner and easier to read.

COMMAND: pivot.plot.area(ax=ax, alpha=0.78, linewidth=0)
WHY: Draws the stacked area chart.
     alpha=0.78 allows slight transparency between stacked areas.
     linewidth=0 removes the border line between areas — cleaner look.

COMMAND: ax.violinplot(data, showmedians=True, widths=0.7)
WHY: Creates violin plots for the salary-by-work-style chart.
     showmedians=True draws a line at the median inside each violin.
     widths=0.7 makes violins slightly narrower so they don't touch.

COMMAND: cbar = fig.colorbar(sc, ax=ax, shrink=0.75)
WHY: Adds a colour legend to scatter/bubble charts that use colour to
     encode a numeric variable (salary level, year).
     shrink=0.75 makes it 75% of the chart height so it's proportional.

COMMAND: plt.setp(cbar.ax.yaxis.get_ticklabels(), color=TEXT)
WHY: Overrides the colorbar tick label colour to white/light.
     Without this, labels default to black and become invisible on dark backgrounds.

SECTION 4 — STREAMLIT COMMANDS EXPLAINED (app.py)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

COMMAND: st.set_page_config(layout="wide", initial_sidebar_state="expanded")
WHY: layout="wide" uses the full browser width instead of the narrow
     centred column Streamlit defaults to.
     initial_sidebar_state="expanded" shows the filters sidebar on load
     instead of requiring users to click to open it.

COMMAND: @st.cache_data(show_spinner="...")
WHY: Caches the result of load_and_clean() after the first run.
     Without this, Pandas re-reads and re-processes the 152MB CSV on
     EVERY filter change — making the dashboard freeze for 10s each time.
     With cache, the data loads once and all interactions are instant.

COMMAND: st.markdown("""...""", unsafe_allow_html=True)
WHY: Renders raw HTML inside Streamlit.
     unsafe_allow_html=True is required to inject custom HTML tags.
     Used for the header title, custom CSS, chart labels, and footer.
     Without it, Streamlit would display the HTML as raw text.

COMMAND: st.multiselect("label", options=all_ages, default=all_ages, label_visibility="collapsed")
WHY: Creates the multi-select filter dropdowns.
     options= defines all available choices.
     default= pre-selects all options (show all data on load).
     label_visibility="collapsed" hides the Streamlit label since we use
     our own custom styled HTML label above each filter.

COMMAND: st.slider("label", min_value=0, max_value=400_000, value=(0, 400_000), format="$%d")
WHY: Creates a range slider for salary and experience filters.
     value=(0, 400_000) sets both handles — start and end.
     format="$%d" displays values as "$75000" instead of "75000".

COMMAND: st.selectbox("AI", options=["All","Yes","No, but I plan to soon","No, and I don't plan to"])
WHY: Single-select dropdown for AI adoption filter.
     selectbox is used (not multiselect) because only one AI status
     should be selected at a time for clean segmentation.

COMMAND: st.columns(5)
WHY: Divides the page into 5 equal-width columns side by side.
     Used for KPI metric cards so all 5 fit in one row without stacking.

COMMAND: st.metric("label", value)
WHY: Streamlit's built-in KPI card widget.
     Displays a label above a large bold number.
     Automatically styled and supports optional delta (change indicator).

COMMAND: st.tabs(["Tab1", "Tab2", ...])
WHY: Organises the 10+ charts into named sections without cluttering
     a single scrolling page. Users click tabs to switch between
     Overview / Salaries / Tech Stack / AI & Culture / Global views.

COMMAND: st.pyplot(fig, use_container_width=True)
WHY: Renders a Matplotlib Figure object inside the Streamlit page.
     use_container_width=True makes charts fill the full column width
     responsively instead of rendering at a fixed pixel size.

COMMAND: st.dataframe(df, use_container_width=True, height=320)
WHY: Renders the filtered data as a scrollable, sortable table.
     height=320 limits the table height so it doesn't push charts off screen.

COMMAND: if st.button("↺ RESET ALL FILTERS"):  st.rerun()
WHY: st.button() creates a clickable button.
     st.rerun() forces Streamlit to re-execute the entire script from top,
     which clears all widget states back to their default values —
     effectively resetting every filter.

COMMAND: st.stop()
WHY: Immediately halts script execution.
     Used after the "no data matches filters" warning so the page
     doesn't try to render empty charts and crash.

SECTION 5 — WHY EACH CHART WAS CHOSEN FOR THIS DATASET
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

──────────────────────────────────────────────────────────────────────────────
CHART 1 — PIE CHART: Remote Work Distribution
COLUMN USED: RemoteWork
REASON: The dataset has exactly 3 meaningful work-style categories —
        Remote, Hybrid, and In-Person. These three sum to 100% of
        respondents who answered this question.
        A pie chart is the ideal choice when:
        (a) there are ≤5 categories
        (b) the proportional relationship matters most
        (c) parts-of-a-whole story is the message
        Alternative considered: Donut chart — rejected as slightly less
        readable for a presentation context.
INSIGHT REVEALED: Over 53% work remotely or hybrid post-COVID.

──────────────────────────────────────────────────────────────────────────────
CHART 2 — HISTOGRAM: Salary Distribution
COLUMN USED: ConvertedCompYearly
REASON: Salary is a continuous numeric variable. A histogram shows:
        - The overall SHAPE of the distribution (right-skewed)
        - Where most salaries cluster ($30K–$120K)
        - The long right tail (outliers earning $250K+)
        - Exactly how many people fall in each salary band
        A bar chart of averages would hide all this richness.
        Colour coding by salary zone (Purple=Low, Teal=Mid, Orange=High,
        Red=Very High) adds instant visual parsing.
INSIGHT REVEALED: Tech salary is heavily right-skewed. Most developers
        earn $40K–$90K; very few exceed $200K.

──────────────────────────────────────────────────────────────────────────────
CHART 3 — LINE CHART: Cumulative Experience Distribution
COLUMN USED: YearsCodePro
REASON: A cumulative distribution function (CDF) line chart answers
        the question "What percentage of developers have X or fewer years
        of experience?" — which is far more actionable than a raw histogram.
        The reference lines at 25%/50%/75% show quartile breakpoints.
        A regular histogram of experience would be harder to interpret.
INSIGHT REVEALED: 50% of professional developers have ≤9 years of
        experience, showing the field skews younger.

──────────────────────────────────────────────────────────────────────────────
CHART 4 — BAR CHART: Top Programming Languages
COLUMN USED: LanguageHaveWorkedWith
REASON: This is a categorical ranking question. Horizontal bar chart is
        the GOLD STANDARD for ranked categories because:
        - Long category names (Python, JavaScript, TypeScript...) fit
          naturally on the Y-axis without rotation
        - Bars are easily compared left-to-right
        - Sorted ascending (longest bar at top) draws the eye correctly
        We also show percentage alongside count for richer context.
INSIGHT REVEALED: JavaScript (62%) leads, followed by HTML/CSS (52%)
        and Python (48%). TypeScript overtook Java.

──────────────────────────────────────────────────────────────────────────────
CHART 5 — SCATTER PLOT: Experience vs Salary
COLUMNS USED: YearsCodePro, ConvertedCompYearly, RemoteWork
REASON: Two continuous variables → scatter plot.
        This is the fundamental chart for showing correlation.
        The trend line confirms salary increases with experience.
        Colouring points by RemoteWork adds a 3rd dimension, showing
        whether work style affects salary at the same experience level.
        We sample 4,000 points from 89K to avoid overplotting.
INSIGHT REVEALED: Clear positive correlation. Remote workers cluster
        slightly higher in the $80K–$150K range at 5–15 years experience.

──────────────────────────────────────────────────────────────────────────────
CHART 6 — BOX PLOT: Salary by Age Group
COLUMNS USED: ConvertedCompYearly, Age
REASON: Box plot is the richest single chart for comparing distributions:
        - Median line = typical salary in that age group
        - Box = middle 50% of earners (IQR)
        - Whiskers = range excluding outliers
        - Dots = outliers
        A bar chart showing average salary per age would hide the spread
        and make all groups look more similar than they are.
        Notched boxes show whether medians are statistically different.
INSIGHT REVEALED: Median salary peaks at age 35–44. Spread (IQR)
        also widens with age, showing more inequality among senior devs.

──────────────────────────────────────────────────────────────────────────────
CHART 7 — HEATMAP: Feature Correlation Matrix
COLUMNS USED: YearsCode, YearsCodePro, WorkExp, ConvertedCompYearly,
              AISelect (as binary), RemoteWork (as binary)
REASON: The correlation heatmap is required by the project spec.
        It's the fastest way to see ALL pairwise numeric relationships
        in one view. Colours from red (negative) to green (positive)
        allow instant scanning.
        We use a lower-triangle mask to remove redundant values.
        We add AI-user and Remote flags as binary (0/1) columns so they
        participate in correlations even though they're categorical.
INSIGHT REVEALED: Salary correlates moderately with experience (r≈0.35).
        AI adoption shows weak correlation with experience level —
        both junior and senior developers are adopting it.

──────────────────────────────────────────────────────────────────────────────
CHART 8 — AREA CHART: Developer Type by Age Group
COLUMNS USED: DevType_Primary, Age
REASON: Stacked area chart shows COMPOSITION across a sequence of groups.
        Unlike a bar chart, area emphasises the flowing transition between
        age groups — how the developer workforce composition shifts as
        developers get older (more students at 18–24, more managers at 45+).
        It shows both individual trends (each layer) and cumulative total
        (total height of stack).
INSIGHT REVEALED: Full-stack developers dominate all ages but decline
        relatively after 35. Backend-only and management roles grow with age.

──────────────────────────────────────────────────────────────────────────────
CHART 9 — COUNT PLOT: AI Adoption by Employment Type
COLUMNS USED: AISelect, Employment_Primary
REASON: A count plot (grouped bar chart) shows FREQUENCY of categories.
        This is the correct chart when the question is
        "How many of group X chose option Y?"
        Grouping by employment type lets us compare patterns:
        Do students adopt AI more than full-timers? Do freelancers lead?
        The grouped layout lets us see both within-group and across-group
        comparisons simultaneously.
INSIGHT REVEALED: Full-time employees have the highest absolute AI users,
        but freelancers have a higher proportional adoption rate.

──────────────────────────────────────────────────────────────────────────────
CHART 10 — VIOLIN PLOT: Salary by Work Style
COLUMNS USED: ConvertedCompYearly, RemoteWork
REASON: A violin plot shows the FULL SHAPE of a distribution, not just
        box/whisker summary statistics. It reveals:
        - Are salaries normally distributed or bimodal within each group?
        - Where is the dense cluster of salaries?
        - How do the distribution SHAPES differ (not just medians)?
        A box plot would only show Remote earns slightly more.
        The violin reveals that Remote has a FATTER distribution at
        high salaries — more people earning $150K+, not just a higher median.
INSIGHT REVEALED: Remote work shows a heavier upper tail.
        In-person work has a tighter, more uniform distribution.

──────────────────────────────────────────────────────────────────────────────
CHART BONUS — BUBBLE CHART: Countries vs Median Salary
COLUMNS USED: Country, ConvertedCompYearly
REASON: Bubble chart encodes THREE variables in 2D:
        X-axis  = country rank (by respondent count)
        Y-axis  = median salary
        Bubble size = number of respondents from that country
        Colour = salary level (plasma gradient)
        This is far richer than a simple bar chart of countries.
        It immediately shows that high-respondent countries (India, US)
        are at opposite ends of the salary scale.
INSIGHT REVEALED: The US has both the most respondents AND the highest
        median salary ($130K+). India has many respondents but low median
        ($15K). Switzerland/Australia punch high on salary with fewer devs.

SECTION 6 — UI / DESIGN CHOICES EXPLAINED
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

COLOUR SCHEME — "DevPulse Dark"
  Background:   #05070D  → Near-black with a blue tint, easier on eyes
                           than pure black (#000000)
  Surface:      #0A1020  → Slightly lighter for chart backgrounds —
                           creates visual depth (layers)
  Primary:      #0FFFA8  → Electric mint/teal — energetic, modern,
                           associated with tech/terminal interfaces
  Secondary:    #7B43FF  → Deep violet — contrasts cleanly with mint
  Highlight:    #FF4B6E  → Coral-red — used for reset button, warnings,
                           and the chart type labels (creates urgency/focus)
  Gold:         #FFD166  → Warm gold for median/trend lines —
                           stands out on dark without being harsh

WHY NOT A LIGHT THEME?
  Developer survey data is best consumed by the audience it describes —
  developers, who overwhelmingly prefer dark themes (survey itself shows
  this). A dark theme also makes neon accent colours pop visually.

FONTS
  Space Grotesk  → Primary body font. Geometric, modern, good readability
  JetBrains Mono → Used for labels, KPI values, data text.
                   A monospaced code font reinforces the tech/dev context.

SIDEBAR FILTER DESIGN
  Each filter has:
  1. A tiny uppercase monospace label (FLABEL) in electric mint
  2. The Streamlit widget with label hidden (our custom label replaces it)
  3. Rounded corners and dark background — consistent "card" feel
  4. Hover glow effect on selectboxes (teal outline appears on hover)

RESET BUTTON — why fire gradient?
  The reset button intentionally uses a warm orange-red gradient to
  visually signal "destructive/reset action" — common UI convention.
  The hover animation (rises 3px + glow) makes it feel alive.

KPI CARD HOVER ANIMATION
  Each metric card lifts 4px on hover (translateY(-4px)).
  This is a subtle but premium interaction — it makes the dashboard
  feel like a real product, not a data science notebook.

TABS vs SINGLE PAGE
  10 charts on one page would require enormous scrolling.
  Tabs group charts by theme (Overview / Salary / Tech / AI / Global)
  so users navigate by what they want to know, not by scrolling position.

SECTION 7 — KEY DATA INSIGHTS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. AI ADOPTION (2023 snapshot)
   - 44% of developers are ALREADY using AI tools
   - 26% plan to use them soon
   - Only 30% have no plans — showing AI adoption was majority by mid-2023

2. REMOTE WORK IS THE NEW NORMAL
   - Only 16% of respondents work fully in-person
   - 42% are hybrid, 41% fully remote
   - This is a permanent structural shift post-COVID

3. JAVASCRIPT DOMINATES — BUT PYTHON IS CLOSING THE GAP
   - JavaScript: 62% | Python: 48% | TypeScript: 38%
   - Python's growth is driven by ML/AI — consistent with AI adoption stats

4. SALARY PEAKS AT 35–44 THEN PLATEAUS
   - Median salary: 25–34 = $65K | 35–44 = $90K | 45–54 = $95K
   - Senior developers earn more but the jump flattens past 35

5. EXPERIENCE MATTERS BUT NOT LINEARLY
   - Years of experience and salary correlate (r≈0.35)
   - Significant variance: 10-yr devs range from $30K to $250K+
   - Specialisation and location matter as much as years

6. US SALARY GAP IS EXTREME
   - US median: ~$130K | India median: ~$15K | UK median: ~$65K
   - Same skills, vastly different compensation — geography is the
     biggest salary determinant in this dataset
